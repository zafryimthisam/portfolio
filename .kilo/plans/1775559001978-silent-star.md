# Replace rembg with @imgly/background-removal-node

## Objective
Replace the current rembg background removal implementation with @imgly/background-removal-node to improve output quality by processing images locally without API calls.

## Current Implementation Analysis
- Uses `@remove-background-ai/rembg.js` which requires an API key (`REMBG_API_KEY`) and sends images to external service
- Supports webp and png output formats
- Processes images server-side via Next.js API route `/api/background-remove`
- Returns processed image as buffer with appropriate content-type

## New Implementation Plan

### 1. Dependency Updates
- Remove `@remove-background-ai/rembg.js` from package.json
- Add `@imgly/background-removal-node` to dependencies
- No additional peer dependencies required for Node.js version

### 2. Environment Configuration
- Remove `REMBG_API_KEY` from .env file since no API key is needed

### 3. Code Changes

#### Update `/api/background-remove/route.ts`
- Replace import: `import { rembg } from "@remove-background-ai/rembg.js";` with `import { removeBackground } from "@imgly/background-removal-node";`
- Remove API key validation logic
- Update processing logic:
  - Call `removeBackground(buffer, { output: { format: format === "webp" ? "image/webp" : "image/png" } })`
  - Handle Blob response: convert to ArrayBuffer then Buffer
  - Remove temp file cleanup logic (no longer needed)
- Maintain same input validation and output format support

#### No changes needed to `ImageBackgroundRemover.tsx`
- Component already sends FormData with image and format
- Expects same response format (processed image blob)

### 4. Testing
- Test with various image formats (PNG, JPEG, WEBP)
- Verify output quality improvement
- Test both webp and png output formats
- Ensure error handling works for invalid inputs
- Test file size limits (50MB)

### 5. Deployment Considerations
- First run will download ONNX models (~80MB), may take time
- Models are cached for subsequent runs
- No external API costs or rate limits
- Better privacy as images stay on server

### 6. Potential Issues
- Initial model download may cause timeout on first request - consider preloading in production
- Ensure Node.js version supports Blob (Node 18+)
- Verify output format mapping works correctly

## Benefits
- Improved image quality (no compression from API processing)
- No API key required
- No external service dependency
- Better privacy (images processed locally)
- Cost savings (no API calls)</content>
